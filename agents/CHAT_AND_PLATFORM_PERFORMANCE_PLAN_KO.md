# HARPER 채팅/전반 속도 개선 계획서
작성일: 2026-02-26

## 목적
- 체감 속도(첫 토큰, 응답 완료, 화면 반응성)를 개선한다.
- 채팅 API뿐 아니라 검색/요약/워커까지 포함한 전체 처리시간을 줄인다.
- 코드 구조를 크게 바꾸기 전에, 측정 가능한 빠른 개선부터 적용한다.

## 우선 측정 지표(KPI)
- 채팅 TTFT(Time To First Token): `p50 <= 1.5s`, `p95 <= 3.0s`
- 채팅 응답 완료 시간(E2E): `p50 <= 6s`, `p95 <= 12s`
- API 실패율: `<= 1%`
- 워커 후보 1인당 요약 시간: `p50 <= 1.2s`(캐시 히트 시), `p95 <= 3.5s`
- 사용자 체감 지표: 입력 후 로딩 표시 노출 `<= 150ms`

## 현재 병목 가능 지점(코드 기준)
- `harper_beta/src/app/api/chat/route.ts`
  - 시스템 프롬프트 길이가 길고 모드별로 컨텍스트가 계속 누적됨
  - `attachments`가 길어지면 토큰/직렬화/LLM 처리시간이 급증
  - 자동화 모드에서 `fetchLatestMemory`가 매 요청마다 DB 왕복
  - tool 루프(`MAX_TOOL_LOOPS=3`, `MAX_TOTAL_TOOL_CALLS=4`)로 LLM 라운드트립이 증가
- `harper_worker/summary.py`
  - 후보별 LLM 요약 호출이 반복되어 전체 처리시간이 선형 증가
  - 동일 criteria 재요청 시 재계산 가능성 큼(캐시 부재 시)

## P0 (1~3일) 빠른 개선
- 프롬프트 경량화
  - 모드별 최소 시스템 프롬프트를 분리하고 공통 규칙 중복을 제거
  - 메시지 히스토리를 최근 N턴 + 요약본으로 압축
  - 첨부파일은 원문 전체 대신 선요약 후 요약본만 주입
- tool 호출 보수화
  - URL이 명시된 경우에만 `website_scraping` 허용
  - 기본 루프 횟수/총 호출 예산을 낮춰 불필요한 왕복 차단
- 메모리 조회 비용 절감
  - `fetchLatestMemory` 결과를 짧은 TTL(예: 30~120초) 캐시
  - 메모리 변화가 없는 대화에서는 재조회 생략
- 모델 라우팅 단순화
  - 짧은 질의/단순 질의는 경량 모델, 복잡 질의만 추론형 모델 사용

## P1 (1~2주) 구조 개선
- 토큰 예산 기반 가드레일
  - 요청당 최대 입력 토큰 상한을 두고, 초과 시 자동 축약
  - 시스템/히스토리/첨부 비중을 분리해 예산 관리
- 워커 캐시/배치 최적화
  - `candidate_id + criteria_hash + model_version` 키로 summary 캐시
  - 동일 조건 재검색 시 캐시 우선 사용
  - 워커 동시성은 상한을 두고(예: 4~8), API rate limit 내에서 병렬 처리
- DB/네트워크 최적화
  - 자주 조회되는 컬럼 인덱스 점검(특히 `memory.user_id + last_updated_at`)
  - 내부 API 호출 체인 최소화(가능하면 직접 함수 호출)

## P2 (2~4주) 안정화/운영
- 실시간 성능 계측
  - 단계별 타이밍: `request_parse`, `memory_fetch`, `prompt_build`, `llm_ttft`, `llm_total`, `tool_call`
  - `request_id`로 프론트-API-워커 추적 연결
- 성능 회귀 방지
  - 배포 전 synthetic 시나리오(짧은 질의/긴 첨부/툴 사용) 자동 측정
  - p95 악화 임계치 초과 시 배포 차단 또는 알림

## 채팅 체감속도 개선 체크리스트
- 입력 즉시 로딩 상태 노출(150ms 이내)
- 이전 요청 취소(Abort)로 중복 스트림 제거
- 긴 답변은 스트리밍 우선, 후처리(정돈/포맷)는 비동기 분리
- 에러 fallback은 1회로 제한해 지연 누적 방지

## 워커/검색 전반 개선 체크리스트
- 동일 쿼리 반복 시 캐시 우선
- 후보 요약 사전생성(또는 변경분만 재생성) 전략 검토
- 결과 수가 많을수록 2단계(빠른 1차 필터 + 느린 정밀 요약) 적용

## 적용 순서 권장
1. 계측(타이밍 로그 + 대시보드) 먼저 추가
2. 프롬프트/히스토리/첨부 축약 정책 적용
3. tool 호출 정책 보수화 + 메모리 캐시 도입
4. 워커 summary 캐시 + 병렬도 조정
5. 배포 후 p50/p95 비교로 단계별 효과 검증

## 기대 효과(보수적)
- 채팅 TTFT: 20~40% 단축
- 채팅 총 응답시간: 15~35% 단축
- 워커 재검색 시간: 캐시 히트 시 50% 이상 단축 가능
